# Epic: ESI Phase 1 — Backend Scheduler + Event Bridge

## Overview

Build the Rust refresh supervisor and per-character refresher tasks, emit Tauri events with ESI payloads, and wire the frontend to consume those events into the existing React Query cache. At the end of this phase, notifications fire reliably regardless of window focus. React Query stays in place for live data; the cadence just switches from JS timers to Rust events.

## Architecture Decisions

- **Cancellation**: `tokio-util::CancellationToken` per refresher. Add `tokio-util` to `Cargo.toml`.
- **Poke mechanism**: `tokio::sync::Notify` per refresher (wakes loop immediately without cancelling).
- **Token refresh**: call `auth::ensure_valid_access_token` at the top of each refresh cycle before any ESI call.
- **Error handling**: on ESI 5xx/network error, log the error, continue loop with a 5-minute backoff sleep (don't crash the refresher).
- **Graceful shutdown**: register Tauri `on_window_event` (or `RunEvent::Exit`) to cancel the supervisor and all refreshers on app quit.
- **Zero-character state**: supervisor loops watching for character adds; no panic or busy-wait when empty.
- **Sleep**: `tokio::time::sleep_until(min(expires) + jitter)`, floor 30s, ceiling 1h.
- **Event naming**: `character:{id}:{resource}` — resources: `queue`, `skills`, `location`, `public`, `attributes`, `clones`, `remaps`. Global: `notifications:new`.
- **Event payload types**: generated by **typeshare** (not tauri-typegen). tauri-typegen has a known bug ignoring `#[serde(rename_all)]` on enums, producing wrong variant strings. typeshare reads serde attributes correctly. Event payload structs get `#[typeshare]` attribute; output to `src/generated/events.ts`. tauri-typegen is untouched and continues generating `commands.ts` / `types.ts`.
- **Frontend adapter**: Phase 1 uses `queryClient.setQueryData(queryKey, payload)` on event receipt — no Zustand yet. `refetchInterval` removed from live-data hooks.

## Technical Approach

### Backend Services

**`src-tauri/src/refresh/mod.rs`** (new module):

- `RefreshSupervisor`: holds `HashMap<i64, RefresherHandle>`, spawns/cancels per-character tasks, reacts to character add/remove.
- `RefresherHandle`: wraps `JoinHandle` + `CancellationToken` + `Arc<Notify>` (poke).
- `RefresherTask`: async loop — token refresh → ESI fetches → diff/persist → notification checks → emit events → sleep.

Reuses existing:

- `esi_helpers::get_cached_skill_queue`, `get_cached_character_attributes`, etc.
- `notifications::NotificationProcessor::process_data_updated`
- `auth::ensure_valid_access_token`
- `db` layer for persist

**`src-tauri/src/lib.rs`**: spawn supervisor in `setup()`, wire poke into existing `force_refresh` command path, add shutdown hook.

**New Tauri command**: `get_esi_snapshot(character_id)` — returns current SQLite state for all live resources (used for initial hydration on frontend mount).

### Frontend Components

**`src/lib/esiEvents.ts`** (new): registers listeners for all `character:*:*` events and `notifications:new`, calls `queryClient.setQueryData` with correct query keys.

**`src/main.tsx` or `src/routes/__root.tsx`**: call `bootstrapEsiEvents(queryClient)` on mount.

**Modified hooks**:

- `useOverview.ts`: remove `refetchInterval`
- `useLocationsOverview.ts`: remove `refetchInterval`
- `useSkillQueue.ts`: remove `refetchInterval` default / make caller not pass one

## Implementation Strategy

Tasks can be partially parallelized: Rust backend (tasks 1–5) and frontend event wiring (tasks 6–7) are independent. Hydration command (task 8) bridges both. Integration verification (task 9) is last.

## Task Breakdown Preview

1. Add `tokio-util` + `typeshare` deps; set up `typeshare.toml` config + `pnpm typeshare` script; add `events.ts` to `src/generated/` convention
2. Define event payload structs in Rust with `#[typeshare]` + `#[derive(Serialize)]`; run typeshare to generate `src/generated/events.ts`
3. Create `refresh/mod.rs` with `RefreshSupervisor` + `RefresherHandle` skeleton
4. Implement `RefresherTask` loop: token refresh → ESI fetches (queue, skills, location, public, attributes, clones, remaps) → persist
5. Integrate `NotificationProcessor` into refresher loop; emit `notifications:new` Tauri event
6. Emit Tauri events per resource after each successful fetch (typed against generated payload structs)
7. Wire supervisor into app lifecycle: spawn on setup, poke on force-refresh, cancel on shutdown
8. Add `get_esi_snapshot` hydration command (reads current SQLite state for all live resources)
9. Frontend: `src/lib/esiEvents.ts` — register event listeners (typed with `events.ts`) → `queryClient.setQueryData`; call on app mount
10. Remove `refetchInterval` from `useOverview`, `useLocationsOverview`, `useSkillQueue`; verify notifications fire with window minimized

## Dependencies

- `tokio-util` (new — CancellationToken)
- `typeshare` crate + `typeshare-cli` (new — event payload type generation)
- All existing: `esi_helpers`, `auth::ensure_valid_access_token`, `NotificationProcessor`, sqlx pool, Tauri app handle

## Success Criteria (Technical)

- Notifications fire on schedule with app minimized (manual test + observation).
- No `refetchInterval` on live-data query hooks.
- `queryClient.setQueryData` is the only path updating live-data cache after Phase 1.
- No refresher panics on ESI errors; errors logged and loop continues.
