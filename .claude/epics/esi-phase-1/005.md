# Task: Integrate NotificationProcessor into refresher loop

## Description

After each successful ESI fetch cycle in `RefresherTask`, call the existing `NotificationProcessor::process_data_updated` for each fetched resource type. Emit the `notifications:new` Tauri event when new notification rows are written. This replaces the old path where notifications were checked at command-invocation time on the frontend polling cycle.

## Acceptance Criteria

- [ ] `NotificationProcessor` instantiated (or shared via Tauri state) and called after each fetch cycle
- [ ] `process_data_updated` called for each data type: `"queue"`, `"skills"`, `"location"`, `"attributes"`, `"clones"`, `"remaps"`
- [ ] `notifications:new` Tauri event emitted when notification checker writes new rows; payload includes `character_id` and the new notification row(s)
- [ ] Notification errors are caught and logged; they do not abort the refresh loop
- [ ] Event payload struct matches the `NotificationsNewPayload` typeshare type from Task 002
- [ ] Existing notification behavior (low queue check) still fires correctly — manual test

## Technical Details

Reuse `notifications::NotificationProcessor` and `notifications::NotificationContext` from `src-tauri/src/notifications/mod.rs`. The `NotificationContext` needs `app`, `pool`, and `rate_limits` — the refresher loop already has access to all three.

Emit with:

```rust
app.emit("notifications:new", &payload)?;
```
