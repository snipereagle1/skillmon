# Task: Rewrite get_esi_snapshot — all characters, all enriched data, DB-only

## Context

Current `get_esi_snapshot` returns partial, minimal snapshot for single character — raw skills, raw attributes, raw clone rows, JSON blob for queue. Frontend calls once per character in fan-out loop on mount.

Task replaces with single command returning fully enriched data for all characters. All reads DB-only — no ESI, no network at launch. Uses same enrichment helpers from tasks 001 and 002, ensuring consistent shapes between events and hydration.

## Work

### `src-tauri/src/commands/esi_snapshot.rs`

Delete existing implementation. Write new `get_esi_snapshot` command:

**Return type**: `Vec<CharacterSnapshot>` — one entry per character.

```rust
#[typeshare]
pub struct CharacterSnapshot {
    pub character_id: i64,
    pub character_name: String,
    pub queue: Option<EnrichedQueue>,       // from enrichment::enrich_queue
    pub skills: Option<EnrichedSkills>,     // from enrichment::enrich_skills
    pub attributes: Option<EnrichedAttributes>, // from enrichment::enrich_attributes
    pub clones: Vec<CloneInfo>,              // from enrichment::enrich_clones — same type as ClonesPayload.clones
    pub location: Option<EnrichedLocation>, // from enrichment::enrich_location (DB-only path)
    pub remaps: Vec<Remap>,                 // from db::get_character_remaps
}
```

**Implementation**:

1. `db::get_all_characters` → iterate characters.
2. Per character, call enrichment fns in parallel (`tokio::join!`):
   - `enrichment::enrich_queue(pool, character_id)` — reads cached queue JSON + SDE enrichment.
   - `enrichment::enrich_skills(pool, character_id)` — reads `db::get_character_skills` + SDE.
   - `enrichment::enrich_attributes(pool, character_id)` — reads `db::get_character_attributes` + implants + remaps.
   - `enrichment::enrich_clones(pool, character_id)` — reads DB clones + implant names.
   - `enrichment::enrich_location_db_only(pool, character_id)` — reads last-known location from DB, resolves names DB-only. Returns `None` if no location row yet.
   - `db::get_character_remaps(pool, character_id)`.
3. Collect into `CharacterSnapshot`. Fields `Option` or empty `Vec` where data may not exist (fresh install, no prior ESI fetch).
4. Return `Vec<CharacterSnapshot>` — frontend iterates, populates all store slices.

**Note on `enrich_location_db_only`**: variant of `enrich_location` (task 002) skipping all ESI, resolving names from SQLite only. Supervisor live path uses full `enrich_location` with ESI fallback; snapshot uses DB-only variant.

### Enrichment module

Ensure `enrichment::enrich_location_db_only` exists, callable without ESI client or rate limiter. May require small refactor of location enrichment helper from task 002 to split ESI-required path from DB-only path.

### `src-tauri/src/commands/mod.rs`

Update registered command signature — `get_esi_snapshot` takes no parameters (no `character_id`).

### Type annotations

`CharacterSnapshot` and all nested enriched types must be annotated `#[typeshare]` to appear in `src/generated/types.ts` after task 004.

## Acceptance Criteria

- `invoke('get_esi_snapshot')` no args returns `CharacterSnapshot[]`.
- Each snapshot covers queue, skills, attributes, clones, location, remaps.
- Command completes under 500ms typical hardware (SQLite-only reads).
- Returns empty/null fields rather than error for characters with no prior ESI data.
- All return types annotated `#[typeshare]`.
- `pnpm lint:rust` passes.
