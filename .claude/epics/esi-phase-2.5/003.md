# Task: Rewrite get_esi_snapshot — all characters, all enriched data, DB-only

## Context

The current `get_esi_snapshot` command returns a partial, minimally enriched snapshot for a single character — raw skills, raw attributes, raw clone rows, and a JSON blob for the queue. The frontend calls it once per character in a fan-out loop on mount.

This task replaces it with a single command that returns fully enriched data for all characters in one call. All reads are DB-only — no ESI calls, no network dependency at launch. The command uses the same enrichment helpers introduced in tasks 001 and 002, ensuring consistent shapes between events and hydration.

## Work

### `src-tauri/src/commands/esi_snapshot.rs`

Delete the existing implementation. Write a new `get_esi_snapshot` command:

**Return type**: `Vec<CharacterSnapshot>` — one entry per character.

```rust
#[typeshare]
pub struct CharacterSnapshot {
    pub character_id: i64,
    pub character_name: String,
    pub queue: Option<EnrichedQueue>,       // from enrichment::enrich_queue
    pub skills: Option<EnrichedSkills>,     // from enrichment::enrich_skills
    pub attributes: Option<EnrichedAttributes>, // from enrichment::enrich_attributes
    pub clones: Vec<CloneInfo>,              // from enrichment::enrich_clones — same type as ClonesPayload.clones
    pub location: Option<EnrichedLocation>, // from enrichment::enrich_location (DB-only path)
    pub remaps: Vec<Remap>,                 // from db::get_character_remaps
}
```

**Implementation**:

1. `db::get_all_characters` → iterate characters.
2. For each character, call enrichment functions in parallel (`tokio::join!`):
   - `enrichment::enrich_queue(pool, character_id)` — reads cached queue JSON + SDE enrichment.
   - `enrichment::enrich_skills(pool, character_id)` — reads `db::get_character_skills` + SDE.
   - `enrichment::enrich_attributes(pool, character_id)` — reads `db::get_character_attributes` + implants + remaps.
   - `enrichment::enrich_clones(pool, character_id)` — reads DB clones + implant names.
   - `enrichment::enrich_location_db_only(pool, character_id)` — reads last-known location from DB, resolves names from DB only (no ESI). Returns `None` if no location row exists yet.
   - `db::get_character_remaps(pool, character_id)`.
3. Collect into `CharacterSnapshot`. Fields are `Option` or empty `Vec` where data may not yet exist (fresh install with no prior ESI fetch).
4. Return `Vec<CharacterSnapshot>` — frontend iterates and populates all store slices.

**Note on `enrich_location_db_only`**: this is a variant of `enrich_location` (from task 002) that skips all ESI calls and resolves names purely from SQLite. The supervisor's live path uses the full `enrich_location` with ESI fallback; the snapshot path uses the DB-only variant.

### Enrichment module

Ensure `enrichment::enrich_location_db_only` exists and is callable without an ESI client or rate limiter. This may require a small refactor of the location enrichment helper introduced in task 002 to split the ESI-required path from the DB-only path.

### `src-tauri/src/commands/mod.rs`

Update the registered command signature — `get_esi_snapshot` now takes no parameters (no `character_id`).

### Type annotations

`CharacterSnapshot` and all nested enriched types must be annotated with `#[typeshare]` so they appear in `src/generated/types.ts` after task 004.

## Acceptance Criteria

- `invoke('get_esi_snapshot')` with no arguments returns `CharacterSnapshot[]`.
- Each snapshot entry covers queue, skills, attributes, clones, location, remaps.
- Command completes in under 500ms on typical hardware (SQLite-only reads).
- Returns empty/null fields rather than erroring for characters with no prior ESI data.
- All return types annotated with `#[typeshare]`.
- `pnpm lint:rust` passes.
