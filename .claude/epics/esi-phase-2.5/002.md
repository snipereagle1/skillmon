# Task: Enrich location/clones supervisor payloads + add ship/online/implants to supervisor

## Context

Location event carries only `{ solar_system_id, station_id, structure_id }` â€” raw IDs, no names. Frontend `CharacterLocationOverview` needs solar system name, region name, station/structure name, ship type name, ship name, online status, implants. Ship/online/implants not fetched by supervisor yet.

Clones event carries raw ESI clone data â€” no location names or implant names.

Both arms have same `Ok(None)` starvation bug as queue/skills/attributes.

## Work

### `src-tauri/src/refresh/events.rs`

**Location payload** (`LocationPayload`) â€” replace `LocationData` with fully enriched struct:

```
LocationPayload {
    character_id: i32,
    has_location_scope: bool,
    solar_system_id: i64,
    solar_system_name: String,
    region_name: Option<String>,
    station_id: Option<i64>,
    station_name: Option<String>,
    structure_id: Option<i64>,
    structure_name: Option<String>,
    structure_type_id: Option<i64>,
    ship_type_id: Option<i64>,
    ship_type_name: Option<String>,
    ship_name: Option<String>,
    is_online: Option<bool>,
    is_docked: Option<bool>,
    implants: Vec<ImplantInfo>,
    character_name: String,
}
```

**Clones payload** (`ClonesPayload`) â€” replace `ClonesData` with enriched struct per clone. Canonical clone shape is `CloneInfo`; payload is `Vec<CloneInfo>` (no nested wrapper). `character_id` at payload envelope level, same as all other resources:

```
ClonesPayload {
    character_id: i32,
    clones: Vec<CloneInfo>,
}
CloneInfo {
    id: i64,
    clone_id: Option<i64>,
    name: Option<String>,
    location_type: String,
    location_id: i64,
    location_name: Option<String>,
    is_current: bool,
    implants: Vec<ImplantInfo>,
}
ImplantInfo {
    type_id: i64,
    name: String,
}
```

**Canonical shape note**: `CloneInfo` and `ImplantInfo` are single source of truth â€” used in event payload and `CharacterSnapshot.clones`. Store holds `ClonesPayload` (full envelope). Listeners write `payload` directly; hydration writes `{ character_id, clones: snapshot.clones }` to match same type.

Annotate all new structs with `#[typeshare]`.

### `src-tauri/src/refresh/mod.rs`

**Location arm**:

- Add parallel fetches alongside `get_cached_character_location`: ship (`get_cached_character_ship`), online status (`get_cached_character_online`), implants (`get_cached_character_implants`).
- Name resolution:
  - Solar system: `esi_helpers::get_cached_solar_system_info` â†’ constellation â†’ region (same chain as `location.rs`).
  - Station: `db::get_station` first; if missing, `esi_helpers::get_cached_station_info` â†’ persist to DB. Subsequent refreshes with unchanged `station_id` use DB only.
  - Structure: same pattern â€” `db::get_structure` first, ESI fallback + persist.
  - **Track last-known IDs**: store `last_solar_system_id`, `last_station_id`, `last_structure_id` in per-character loop state. ESI name resolution only fires on ID change.
- Resolve ship type name via `utils::get_type_names`.
- Resolve implant names via `utils::get_type_names`.
- `Ok(None)` arm (location cache hit): read last-known location from `db::get_character_location_cache`, enrich with DB-only name reads (no ESI), emit.

**Clones arm**:

- After fetching clones, read each clone's implants from DB (`db::get_clone_implants`).
- Resolve location names from DB (`db::get_station` / `db::get_structure` by `location_id` and `location_type`).
- Resolve implant names via `utils::get_type_names`.
- `Ok(None)` arm: read `db::get_character_clones`, enrich per-clone from DB, emit.

## Acceptance Criteria

- `character:{}:location` payload includes `solar_system_name`, `region_name`, `station_name`/`structure_name`, `ship_type_name`, `ship_name`, `is_online`, `is_docked`, `implants[]`.
- `character:{}:clones` payload includes per-clone `location_name` and `implants[]` with names.
- Station/structure ESI name resolution fires only on ID change between refresh cycles.
- Cache-warm arms still emit using last-known DB data.
- No-location-scope characters emit `has_location_scope: false` with null fields (no error).
- `pnpm lint:rust` passes.
