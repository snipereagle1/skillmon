# Task: Enrich location/clones supervisor payloads + add ship/online/implants to supervisor

## Context

The location event currently carries only `{ solar_system_id, station_id, structure_id }` — raw IDs with no names. The frontend's `CharacterLocationOverview` needs solar system name, region name, station/structure name, ship type name, ship name, online status, and implants. Ship, online status, and implants are not currently fetched by the supervisor at all.

The clones event carries only raw ESI clone data — no location names or implant names.

Both arms have the same `Ok(None)` starvation bug as queue/skills/attributes.

## Work

### `src-tauri/src/refresh/events.rs`

**Location payload** (`LocationPayload`) — replace `LocationData` with a fully enriched struct:

```
LocationPayload {
    character_id: i32,
    has_location_scope: bool,
    solar_system_id: i64,
    solar_system_name: String,
    region_name: Option<String>,
    station_id: Option<i64>,
    station_name: Option<String>,
    structure_id: Option<i64>,
    structure_name: Option<String>,
    structure_type_id: Option<i64>,
    ship_type_id: Option<i64>,
    ship_type_name: Option<String>,
    ship_name: Option<String>,
    is_online: Option<bool>,
    is_docked: Option<bool>,
    implants: Vec<ImplantInfo>,
    character_name: String,
}
```

**Clones payload** (`ClonesPayload`) — replace `ClonesData` with enriched struct per clone. The canonical clone shape is `CloneInfo`; the payload is `Vec<CloneInfo>` (no nested wrapper). `character_id` is carried at the payload envelope level, same as all other resources:

```
ClonesPayload {
    character_id: i32,
    clones: Vec<CloneInfo>,
}
CloneInfo {
    id: i64,
    clone_id: Option<i64>,
    name: Option<String>,
    location_type: String,
    location_id: i64,
    location_name: Option<String>,
    is_current: bool,
    implants: Vec<ImplantInfo>,
}
ImplantInfo {
    type_id: i64,
    name: String,
}
```

**Canonical shape note**: `CloneInfo` and `ImplantInfo` are the single source of truth used in both the event payload and `CharacterSnapshot.clones`. The store holds `ClonesPayload` (the full envelope). Listeners write `payload` directly; hydration writes `{ character_id, clones: snapshot.clones }` to match the same type.

Annotate all new structs with `#[typeshare]`.

### `src-tauri/src/refresh/mod.rs`

**Location arm**:

- Add parallel fetches alongside `get_cached_character_location`: ship (`get_cached_character_ship`), online status (`get_cached_character_online`), implants (`get_cached_character_implants`).
- Name resolution strategy:
  - Solar system: `esi_helpers::get_cached_solar_system_info` → constellation → region (same chain as `location.rs`).
  - Station: `db::get_station` first; if missing, `esi_helpers::get_cached_station_info` → persist to DB. On subsequent refreshes where `station_id` hasn't changed, DB read is sufficient.
  - Structure: same pattern — `db::get_structure` first, ESI fallback + persist.
  - **Track last-known IDs**: store `last_solar_system_id`, `last_station_id`, `last_structure_id` in the per-character loop state. Only call ESI name resolution when an ID changes.
- Resolve ship type name via `utils::get_type_names`.
- Resolve implant names via `utils::get_type_names`.
- `Ok(None)` arm (location cache hit): read last-known location from `db::get_character_location_cache` (or equivalent stored row), enrich with DB-only name reads (no ESI), emit.

**Clones arm**:

- After fetching clones, read each clone's implants from DB (`db::get_clone_implants`).
- Resolve location names from DB (`db::get_station` / `db::get_structure` by `location_id` and `location_type`).
- Resolve implant names via `utils::get_type_names`.
- `Ok(None)` arm: read `db::get_character_clones`, enrich per-clone from DB, emit.

### Enrichment helpers

Add to the shared enrichment module (started in task 001):

- `enrich_location(pool, client, character_id, rate_limits, last_ids) -> EnrichedLocation`
- `enrich_clones(pool, character_id) -> EnrichedClones`

`enrich_location` takes `last_ids` to gate ESI name resolution calls.

## Acceptance Criteria

- `character:{}:location` event payload includes `solar_system_name`, `region_name`, `station_name`/`structure_name`, `ship_type_name`, `ship_name`, `is_online`, `is_docked`, `implants[]`.
- `character:{}:clones` event payload includes per-clone `location_name` and `implants[]` with names.
- Station/structure ESI name resolution only fires when the relevant ID changes between refresh cycles.
- When cache is warm, both arms still emit using last-known DB data.
- Characters without the location scope emit `has_location_scope: false` with null fields (no error).
- `pnpm lint:rust` passes.
