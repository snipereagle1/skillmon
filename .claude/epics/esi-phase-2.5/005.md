# Task: Rewrite esiEvents.ts as pure setters + rewrite mount hydration in \_\_root.tsx

## Context

`src/lib/esiEvents.ts` currently ignores event payloads and calls enrichment commands after each event fires. With enriched payloads now delivered by the supervisor (tasks 001, 002), listeners can become pure setters — receive payload, write to store, done. Zero async, zero network.

`src/routes/__root.tsx` hydration calls 5+ commands per character in a fan-out loop. The new `get_esi_snapshot` (task 003) replaces this with a single `invoke()` call returning all characters at once.

## Work

### `src/lib/esiEvents.ts`

Rewrite every listener to a pure setter. Each listener receives the typed payload (now from `@/generated/types` or `@/generated/events`) and calls the matching store setter directly.

Remove all imports of deleted commands. Remove the `/* eslint-disable no-await-in-loop */` directive (no longer needed — no async in listeners).

New shape:

```typescript
const unlistenQueue = await listen<QueuePayload>(
  `character:${characterId}:queue`,
  ({ payload }) => useEsiStore.getState().setQueue(characterId, payload)
);

const unlistenSkills = await listen<SkillsPayload>(
  `character:${characterId}:skills`,
  ({ payload }) => useEsiStore.getState().setSkills(characterId, payload)
);

// ... same pattern for attributes, location, clones
```

The `notifications:new` listener is unchanged (it invalidates TanStack Query, not the ESI store).

Update `useEsiStore` setter types (`setQueue`, `setSkills`, etc.) to accept the new enriched payload types from `@/generated/types` rather than the old command return types. The store shape itself (`queues`, `skills`, etc.) now holds the enriched event payload shapes directly.

### `src/stores/esiStore.ts`

Update the type parameters for each store slice to match the new enriched payload types:

- `queues: Record<number, ResourceSlice<QueuePayload>>`
- `skills: Record<number, ResourceSlice<SkillsPayload>>`
- `attributes: Record<number, ResourceSlice<AttributesPayload>>`
- `locations: Record<number, ResourceSlice<LocationPayload>>`
- `clones: Record<number, ResourceSlice<ClonePayload>>`

Remaps slice type is unchanged (still driven by command).

### `src/routes/__root.tsx`

Replace the per-character hydration fan-out with a single `invoke<CharacterSnapshot[]>('get_esi_snapshot')` call:

```typescript
const snapshots = await invoke<CharacterSnapshot[]>('get_esi_snapshot');
for (const snapshot of snapshots) {
  const id = snapshot.character_id;
  if (snapshot.queue) store.setQueue(id, snapshot.queue);
  if (snapshot.skills) store.setSkills(id, snapshot.skills);
  if (snapshot.attributes) store.setAttributes(id, snapshot.attributes);
  if (snapshot.clones.length > 0) store.setClones(id, snapshot.clones);
  if (snapshot.location) store.setLocation(id, snapshot.location);
  if (snapshot.remaps.length > 0) store.setRemaps(id, snapshot.remaps);
}
```

Per-slice error handling: wrap each `set*` call — if `invoke` itself throws, set `lastError` on all slices for all characters so `isLoading` flips to false and an error state renders instead of a permanent spinner.

The `getAccountsAndCharacters` call is **kept** as a prerequisite, but only to obtain character IDs for `bootstrapEsiEvents`. It is not used to drive the hydration loop (that comes from the snapshot). The hydration sequence becomes:

```typescript
const accountsData = await getAccountsAndCharacters(); // lightweight — needed for character IDs
const characterIds = [
  ...accountsData.unassigned_characters,
  ...accountsData.accounts.flatMap((a) => a.characters),
].map((c) => c.character_id);

try {
  const snapshots = await invoke<CharacterSnapshot[]>('get_esi_snapshot');
  // populate store from snapshots...
} catch (err) {
  // set lastError on all known character slices so isLoading flips false
  for (const id of characterIds) {
    setError('queues', id, String(err));
    setError('skills', id, String(err));
    // etc.
  }
}

cleanup = await bootstrapEsiEvents(queryClient, characterIds);
```

This preserves the guarantee that listeners are registered for all known characters regardless of snapshot success or failure.

Remove all other imports of deleted commands from `__root.tsx`.

`bootstrapEsiEvents` calling convention is unchanged.

## Acceptance Criteria

- `esiEvents.ts` contains zero `invoke()` calls and zero `async` listener callbacks.
- All listener payloads are typed using the generated event payload types.
- Mount hydration calls `invoke('get_esi_snapshot')` exactly once (no per-character loop).
- If `get_esi_snapshot` fails, all store slices for all characters have `lastError` set — no permanent spinners.
- `pnpm typecheck` passes.
- `pnpm lint` passes (no unused imports from deleted commands).
