# Task: Migrate Attributes, Clones, and Remaps Hooks to Zustand

## Description

Migrate three remaining live-data read hooks to Zustand in a single task: `useAttributes`, `useClones`, and `useCharacterRemaps` (the read hook in `useRemaps.ts`). Update `esiEvents.ts` for all three resources. Remove the final Phase 1 RQ adapters.

The mutation hooks in `useRemaps.ts` (`useSaveRemap`, `useDeleteRemap`) are **not changed** — they stay as RQ mutations and continue invalidating query cache as-is.

## Acceptance Criteria

- [ ] `useAttributes(characterId)` reads from `state.attributes[characterId]?.data` — no `useQuery`
- [ ] `useClones(characterId)` reads from `state.clones[characterId]?.data` — no `useQuery`
- [ ] `useCharacterRemaps(characterId)` reads from `state.remaps[characterId]?.data` — no `useQuery`
- [ ] `esiEvents.ts` has store setter calls for attributes, clones, remaps — no `queryClient.setQueryData` for any of these
- [ ] All three Phase 1 RQ adapters removed
- [ ] `useSaveRemap` and `useDeleteRemap` mutations unchanged
- [ ] Consumer components unchanged
- [ ] `pnpm typecheck` passes, `pnpm test` passes

## Technical Details

`useCharacterRemaps` is one export in `useRemaps.ts` alongside the mutation hooks. Only migrate the read function — keep the file intact for the mutations.

After mutations succeed (save/delete remap), they call `queryClient.invalidateQueries` for `queryKeys.remaps.character` and `queryKeys.attributes`. Since those query keys no longer back the live hooks, remove those invalidations and instead poke the Rust supervisor (if a poke command exists from Phase 1) or accept that the next scheduled refresh will update the store. Document the decision.

Verify call sites for all three: `grep -r "useAttributes\|useClones\|useCharacterRemaps" src/`.
