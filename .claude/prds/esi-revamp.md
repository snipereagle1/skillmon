# PRD: ESI Revamp — Backend-Driven Refresh

## Executive Summary

ESI data refresh currently runs on frontend JS timers (TanStack Query `refetchInterval`). OS timer throttling when the app is backgrounded causes missed/late notifications. Moving the refresh clock to Rust fixes this, and migrating live-data state to Zustand simplifies the frontend data flow.

## Problem Statement

- Frontend timers are throttled/paused when the renderer is backgrounded.
- Notifications (low queue, etc.) are checked at refresh time in Rust — so throttled refreshes = missed notifications.
- The app is resident in the system tray and Rust is fully capable of running refresh independently.

## User Stories

- As a user with the app minimized, I expect low-queue notifications to fire on schedule.
- As a user, I expect the UI to show fresh ESI data as soon as I bring the app to the foreground.
- As a developer, I want a single, consistent data flow: Rust pushes live state, React renders it.

## Functional Requirements

### Backend

- Refresh supervisor spawned at app launch; manages one task per character.
- Per-character refresher loop: fetch ESI → diff → persist → notify → emit events → sleep until `expires`.
- Refreshers are pokeable (wake immediately) for manual refresh, SSO re-auth, etc.
- Refreshers are cancellable via `tokio-util::CancellationToken`.
- Supervisor reacts to character add/remove without app restart.
- Refresher calls `ensure_valid_access_token` before each cycle (handles OAuth token refresh).
- On ESI error (5xx/network): log, emit error event with `lastError` metadata, backoff 5 min, resume.
- Graceful shutdown: app-close Tauri lifecycle hook cancels all refreshers cleanly.
- Zero-character state: supervisor starts as no-op and waits for first character to be added.
- Sleep clamped: 30s floor, 1h ceiling.

### Event Bridge

- Full resource values as payloads (not "dirty" signals).
- Naming: `character:{id}:{resource}` (e.g., `character:12345:queue`).
- Resources in scope: `queue`, `skills`, `location`, `public`, `attributes`, `clones`, `remaps`.
- Global: `notifications:new` (includes `character_id` in payload).
- **Event payload types are generated by typeshare**, not tauri-typegen. tauri-typegen has a known bug where it ignores `#[serde(rename_all)]` on enums, producing incorrect variant strings. typeshare correctly reads serde attributes and generates accurate TS types.
- Event payload Rust structs are annotated with `#[typeshare]` and output to `src/generated/events.ts`.
- tauri-typegen is retained for command bindings (`commands.ts`, `types.ts`). The two tools are non-overlapping: typeshare owns `events.ts`, tauri-typegen owns everything else in `src/generated/`.

### Frontend

- Zustand `useEsiStore`: per-resource slices keyed by `character_id`, each with `data`, `lastUpdatedAt`, `lastError`.
- Single event-listener bootstrap on app mount.
- Custom hooks (`useQueue`, `useSkillQueue`, etc.) read from store via selectors.
- Hook signatures match existing RQ hooks where possible (no consumer component changes).
- TanStack Query retained for: mutations, SDE/static reads, settings, one-shot fetches.
- Initial hydration: existing Tauri commands hydrate store from SQLite on mount before first ESI push.

## Non-Functional Requirements

- Memory: N tasks per character; acceptable for typical use (1–10 chars).
- Launch: all refreshers fire immediately on start (existing rate-limit awareness handles burst).
- Tauri event payload size: full queues/skill lists are small; verify no Tauri payload limit issues.
- Phase 1–2 dual-system window: some hooks Zustand-backed, some RQ-backed; no hook gets two listeners.

## Success Criteria

- Notifications fire on schedule with app minimized, on different desktop, or out of focus — all supported platforms.
- Refresh cadence per character aligns to ESI `expires`, not a fixed JS interval.
- Live data hooks read exclusively from Zustand; no `refetchInterval` for ESI-derived data.
- React Query usage limited to mutations, SDE/static reads, settings.

## Constraints & Assumptions

- `ensure_valid_access_token` in `src-tauri/src/auth/oauth.rs` handles OAuth token refresh and must be called at the start of each refresher cycle.
- `tokio-util` not currently in `Cargo.toml`; must be added.
- `typeshare` crate not currently in `Cargo.toml`; must be added. `typeshare-cli` must be available in the dev environment.
- tauri-typegen is retained for command bindings and must not be replaced for that purpose.
- `src/generated/events.ts` is auto-generated by typeshare — never edit by hand. `pnpm typeshare` script regenerates it.
- Existing `NotificationProcessor` and `esi_helpers` are reused as-is.
- Migration is incremental: each phase ships independently.

## Out of Scope (v1)

- Per-skill completion timers (precise `sleep_until(finish_time)` notifications).
- Sleep/wake OS event handling beyond "loop resumes and refreshes on wake."
- CRDT/multi-machine sync.
- Refactoring SDE or static-data flows.
- Refactoring mutation flows.
- Adding new ESI endpoints or notification types.

## Dependencies

- `tokio-util` crate (CancellationToken).
- `typeshare` crate + `typeshare-cli` (event payload type generation).
- Existing: `NotificationProcessor`, `esi_helpers`, `ensure_valid_access_token`, sqlx pool, Tauri app handle.
- Frontend: Zustand (already in stack), TanStack Query retained.
